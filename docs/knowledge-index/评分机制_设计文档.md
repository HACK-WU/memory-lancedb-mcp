## 1. 需求背景 & 目标

### 1.1 背景

知识索引系统需要判断哪些索引、Relation、关键词是热门的，以便优先返回给 AI Agent。当前设计缺乏系统化的评分机制，无法准确反映使用频率。

### 1.2 目标

- 目标 1：建立基于使用频率的评分机制，准确反映数据的热门程度
- 目标 2：实现冷热分区机制，优化查询效率
- 目标 3：支持索引、Relation、关键词的统一评分管理
- 目标 4：提供热门索引优先返回机制

## 2. 名词术语表

| 术语 | 含义 | 易混淆点 |
|------|------|---------|
| **评分** | 数据的使用频率指标，值越大表示越常被命中 | 不是简单的计数，是基于频率的动态评分 |
| **冷热分区** | 根据评分将数据分为热区、常温区、冷区三个等级 | 不是物理分区，是逻辑分区 |
| **热区** | 评分最高的数据，优先返回给 Agent | 不是固定数量，根据评分动态调整 |
| **常温区** | 评分中等的数据，次优先返回 | 不是永久存储，可能被转移到冷区 |
| **冷区** | 评分最低的数据，最后返回或可能被删除 | 不是归档，可能被直接删除 |
| **评分衰减** | 长时间未使用的数据评分下降 | 不是重置，是逐渐衰减 |

## 3. 评分机制设计

### 3.1 设计目标

**核心需求**：
1. **经常用的评分高**：使用频率越高，评分越高
2. **不经常用的评分低**：长时间不用，评分自动衰减
3. **防止刷分**：短时间内频繁调用不重复计分
4. **快速响应新兴热门**：最近频繁使用的内容能快速进入热区

**解决的关键问题**：
- 历史霸占热区：历史高分内容长期占据热区
- 新兴热门响应慢：新内容无法快速进入热区
- 冷启动问题：新内容需要很长时间才能累积到高分

### 3.2 评分算法：基于使用密度

评分由三部分组成：
1. **基础评分**：基于使用密度（平均使用间隔）
2. **时间衰减**：长时间不用自动降分
3. **活跃度加成**：最近使用过的内容获得额外加成

#### 3.2.1 基础评分算法

```javascript
/**
 * 计算基础评分（基于使用密度）
 * 
 * 核心公式：
 *   densityScore = 100 / (1 + avgIntervalMinutes / halfLifeMinutes)
 * 
 * 其中：
 *   avgIntervalMinutes = 最近N次使用的平均间隔（分钟）
 *   halfLifeMinutes = 半衰期，默认30分钟
 *   密度越高（间隔越短），评分越高
 */

const CONFIG = {
  minRecordInterval: 5,           // 最小记录间隔（分钟），小于此值不重复记录
  maxRecordCount: 10,             // 最多保留的使用记录数
  halfLifeMinutes: 30,            // 半衰期（分钟），间隔为此值时得分为50
};

/**
 * 记录一次使用
 * @param {number[]} lastUsedTimes - 最近使用时间戳数组
 * @param {number} now - 当前时间戳
 * @returns {number[]} 更新后的时间戳数组
 */
function recordUse(lastUsedTimes, now) {
  const minInterval = CONFIG.minRecordInterval * 60 * 1000;
  
  // 如果与上次记录间隔小于最小记录间隔，忽略本次记录（防止刷分）
  if (lastUsedTimes.length > 0) {
    const lastTime = lastUsedTimes[lastUsedTimes.length - 1];
    if (now - lastTime < minInterval) {
      return lastUsedTimes;
    }
  }
  
  // 添加新记录，保留最近N次
  const times = [...lastUsedTimes, now];
  if (times.length > CONFIG.maxRecordCount) {
    times.shift();
  }
  return times;
}

/**
 * 计算基础评分（基于使用密度）
 * @param {number[]} lastUsedTimes - 最近使用时间戳数组
 * @returns {number} 基础评分（0-100）
 */
function calculateDensityScore(lastUsedTimes) {
  // 没有足够的使用记录，返回最低分
  if (lastUsedTimes.length < 2) {
    return 1;
  }

  // 计算加权平均使用间隔（越近的间隔权重越大）
  let totalWeightedInterval = 0;
  let totalWeight = 0;

  for (let i = 1; i < lastUsedTimes.length; i++) {
    const intervalMinutes = (lastUsedTimes[i] - lastUsedTimes[i - 1]) / (60 * 1000);
    // 越新的间隔权重越大，使用指数权重
    const weight = Math.exp(i / lastUsedTimes.length);
    totalWeightedInterval += intervalMinutes * weight;
    totalWeight += weight;
  }
  
  const avgIntervalMinutes = totalWeightedInterval / totalWeight;

  // 计算使用密度得分
  // avgInterval越小，得分越高
  // avgInterval = halfLife时，得分为50
  // avgInterval → 0时，得分 → 100
  // avgInterval → ∞时，得分 → 0
  return Math.round(
    100 / (1 + avgIntervalMinutes / CONFIG.halfLifeMinutes)
  );
}
```

#### 3.2.2 边界衰减机制

**设计目标**：避免对所有内容实时计算衰减，只在内容进入热区时触发边界衰减。

**衰减触发条件**：
- 新内容评分 > 热区最低分
- 新内容要进入热区时触发

**衰减流程**：
1. 保存常温区最高分 `origin_max`
2. 常温区最高分 -10
3. 热区最低分 = `origin_max`（衰减到常温区最高分）
4. 热区最高分 -10
5. 新内容进入热区

**边界衰减算法**：
```javascript
/**
 * 边界衰减：当新内容要进入热区时触发
 * @param {Array} hotItems - 热区内容列表（按评分降序）
 * @param {Array} warmItems - 常温区内容列表（按评分降序）
 * @param {number} newScore - 新内容评分
 * @returns {Object} 衰减后的分区状态
 */
function boundaryDecay(hotItems, warmItems, newScore) {
  // 检查是否需要触发衰减
  if (hotItems.length === 0 || newScore <= hotItems[hotItems.length - 1].score) {
    return { hotItems, warmItems, triggered: false };
  }
  
  // 步骤1：保存常温区最高分
  const originMax = warmItems.length > 0 ? warmItems[0].score : 0;
  
  // 步骤2：常温区最高分 -10
  if (warmItems.length > 0) {
    warmItems[0].score = Math.max(0, warmItems[0].score - 10);
  }
  
  // 步骤3：热区最低分衰减到原常温区最高分
  const hotMinItem = hotItems[hotItems.length - 1];
  hotMinItem.score = originMax;
  
  // 步骤4：热区最高分 -10
  if (hotItems.length > 0) {
    hotItems[0].score = Math.max(0, hotItems[0].score - 10);
  }
  
  // 步骤5：新内容进入热区（由调用方处理）
  return { hotItems, warmItems, triggered: true, originMax };
}
```

**边界衰减的优势**：
- 无需对所有内容计算衰减，只在边界触发
- 计算量 O(1)，性能最优
- 自动流动：热区最低分流入常温区
- 分数平衡：热区和常温区最高分都会下降

#### 3.2.3 活跃度加成算法

```javascript
/**
 * 计算活跃度加成系数
 * @param {number} lastUsedTime - 最后使用时间戳
 * @param {number} now - 当前时间戳
 * @returns {number} 加成系数（1.0-2.0）
 */

const ACTIVITY_CONFIG = {
  recentHours: 48,  // "最近活跃"的时间窗口（小时）
};

function calculateActivityBonus(lastUsedTime, now) {
  const hoursSinceLastUse = (now - lastUsedTime) / (60 * 60 * 1000);
  
  // 超过活跃时间窗口，无加成
  if (hoursSinceLastUse > ACTIVITY_CONFIG.recentHours) {
    return 1.0;
  }
  
  // 在活跃时间窗口内，根据时间给予加成
  if (hoursSinceLastUse < 1) {
    return 2.0;    // 1小时内：评分×2
  } else if (hoursSinceLastUse < 6) {
    return 1.8;    // 6小时内：评分×1.8
  } else if (hoursSinceLastUse < 24) {
    return 1.5;    // 24小时内：评分×1.5
  } else {
    return 1.2;    // 48小时内：评分×1.2
  }
}
```

#### 3.2.4 最终评分计算

```javascript
/**
 * 计算最终评分
 * @param {number[]} lastUsedTimes - 最近使用时间戳数组
 * @param {number} now - 当前时间戳
 * @returns {number} 最终评分（0-200，因为活跃度加成可能超过100）
 */
function calculateFinalScore(lastUsedTimes, now) {
  // 1. 计算基础评分（基于使用密度）
  const baseScore = calculateDensityScore(lastUsedTimes);
  
  // 2. 计算活跃度加成
  const lastUsedTime = lastUsedTimes[lastUsedTimes.length - 1];
  const activityBonus = calculateActivityBonus(lastUsedTime, now);
  
  // 3. 最终评分 = 基础分 × 活跃度加成
  // 注意：衰减由边界衰减机制处理，不在计算评分时进行
  return Math.round(baseScore * activityBonus);
}
```

### 3.3 评分变化示例

| 使用模式 | 密度评分 | 活跃度加成 | 最终评分 | 说明 |
|---------|---------|-----------|---------|------|

| 每15分钟用一次（正常，2小时前） | 67 | 1.8 | **121** | 高频使用，评分高 |
| 每30分钟用一次（半衰点，1小时前） | 50 | 1.8 | **90** | 中频使用，评分中等 |
| 每1小时用一次（偶尔，3小时前） | 33 | 1.2 | **40** | 低频使用，评分较低 |
| 每4小时用一次（低频，1天前） | 11 | 1.2 | **13** | 低频使用，评分低 |
| 每天用一次（低频，2天前） | 2 | 1.2 | **2** | 很低频，评分很低 |

- 活跃度加成基于48小时时间窗口

### 3.4 防刷分机制

```javascript
/**
 * 防刷分检测
 * @param {number[]} lastUsedTimes - 最近使用时间戳
 * @param {number} now - 当前时间戳
 * @returns {boolean} 是否为有效使用
 */
function isValidUse(lastUsedTimes, now) {
  if (lastUsedTimes.length === 0) return true;
  
  const lastTime = lastUsedTimes[lastUsedTimes.length - 1];
  const minInterval = CONFIG.minRecordInterval * 60 * 1000; // 5分钟
  
  return (now - lastTime) >= minInterval;
}
```

### 3.6 评分初始化

- **新建数据**：评分初始化为 0，lastUsedTimes 为空数组
- **导入数据**：评分初始化为 0，lastUsedTimes 为空数组
- **手动设置**：可通过管理脚本手动设置评分和 lastUsedTimes

## 4. 冷热分区机制（混合策略 + 新兴热区）

### 4.1 设计目标

**解决的核心问题**：
1. **历史霸占热区**：历史高分内容长期占据热区，新内容难以进入
2. **新兴热门响应慢**：新内容无法快速进入热区
3. **热区空置**：如果所有内容评分都不高，热区可能为空
4. **衰减计算开销**：避免对所有内容实时计算衰减

**混合策略思路**：
- **新兴热区**：为最近48小时内使用过的内容保留席位
- **历史热区**：为评分高的历史内容保留席位
- **边界衰减**：只在内容进入热区时触发衰减，计算量 O(1)

### 4.2 三区设计

**热区分为两部分**：
1. **新兴热区**：最近48小时内使用过的内容，有保留席位（默认10个）
2. **历史热区**：评分高的历史内容，按相对排名分配

**分区结构**：
```
热区 = 新兴热区 + 历史热区
常温区 = 中等评分内容
冷区 = 低评分内容
```

### 4.3 分区算法

```javascript
/**
 * 混合分区算法（含新兴热区）
 * 
 * 分区规则：
 * 1. 过滤：只考虑达到最低分数阈值的内容
 * 2. 识别新兴热门：最近48小时内使用过的内容
 * 3. 分配新兴热区：保留席位给新兴热门内容
 * 4. 分配历史热区：按评分排序，填充剩余热区席位
 * 5. 分配常温区和冷区
 * 
 * @param {Array} items - 数据列表（包含 score, lastUsedTime 等字段）
 * @param {Object} config - 分区配置
 * @param {number} now - 当前时间戳
 * @returns {Object} 分区结果
 */

const PARTITION_CONFIG = {
  minScoreThreshold: 10,      // 最低进入热区分数
  hotPercent: 0.3,            // 热区占比（前30%）
  warmPercent: 0.5,           // 常温区占比（中间50%）
  reservedEmerging: 10,       // 为新兴热门保留的席位
  recentHours: 48,            // "新兴热门"的时间窗口（小时）
  minHotCount: 1,             // 最少热区数量
};

function hybridPartition(items, now, config = PARTITION_CONFIG) {
  // 步骤1：计算每个内容的最终评分
  const itemsWithScore = items.map(item => {
    const finalScore = calculateFinalScore(item.lastUsedTimes, now);
    return {
      ...item,
      finalScore,
      lastUsedTime: item.lastUsedTimes[item.lastUsedTimes.length - 1]
    };
  });
  
  // 步骤2：过滤达到最低分数阈值的内容
  const qualified = itemsWithScore.filter(
    item => item.finalScore >= config.minScoreThreshold
  );
  
  // 步骤3：识别新兴热门内容（最近48小时内使用过）
  const recentThreshold = config.recentHours * 60 * 60 * 1000;
  const emergingItems = qualified.filter(
    item => (now - item.lastUsedTime) < recentThreshold
  );
  
  // 步骤4：按最终评分排序（用于历史高分内容）
  qualified.sort((a, b) => b.finalScore - a.finalScore);
  
  // 步骤5：分配热区席位
  const hot = [];
  const warm = [];
  const cold = [];
  
  // 计算热区总席位数
  const totalHotSeats = Math.max(
    config.minHotCount,
    Math.ceil(qualified.length * config.hotPercent)
  );
  
  // 先分配新兴热区席位（保留席位）
  const emergingHotSeats = Math.min(config.reservedEmerging, emergingItems.length);
  for (let i = 0; i < emergingHotSeats && i < emergingItems.length; i++) {
    hot.push(emergingItems[i]);
  }
  
  // 再分配历史热区席位（填充剩余热区席位）
  const historicalHotSeats = totalHotSeats - hot.length;
  for (let i = 0; i < historicalHotSeats && i < qualified.length; i++) {
    const item = qualified[i];
    // 避免重复添加已经在热区的新兴热门内容
    if (!hot.includes(item)) {
      hot.push(item);
    }
  }
  
  // 确保最少热区数量
  while (hot.length < config.minHotCount && hot.length < qualified.length) {
    const nextItem = qualified.find(item => !hot.includes(item) && !warm.includes(item));
    if (nextItem) {
      hot.push(nextItem);
    } else {
      break;
    }
  }
  
  // 分配常温区和冷区
  const remaining = qualified.filter(
    item => !hot.includes(item)
  );
  
  const warmCount = Math.ceil(qualified.length * config.warmPercent);
  warm.push(...remaining.slice(0, warmCount));
  cold.push(...remaining.slice(warmCount));
  
  // 处理未达标的内容（分数低于阈值）
  const unqualified = itemsWithScore.filter(
    item => item.finalScore < config.minScoreThreshold
  );
  cold.push(...unqualified);
  
  return {
    hot,
    warm,
    cold,
    stats: {
      total: items.length,
      qualified: qualified.length,
      hotCount: hot.length,
      warmCount: warm.length,
      coldCount: cold.length,
      emergingCount: emergingItems.length,
      emergingHotCount: emergingHotSeats,
      historicalHotCount: hot.length - emergingHotSeats,
      thresholds: {
        minScore: config.minScoreThreshold,
        hotMinScore: hot.length > 0 ? hot[hot.length - 1].finalScore : 0,
        warmMinScore: warm.length > 0 ? warm[warm.length - 1].finalScore : 0,
      }
    }
  };
}
```

### 4.4 分区转移规则

**升级流程**（新内容进入热区）：
1. 新内容评分 > 热区最低分
2. 触发边界衰减：
   - 保存常温区最高分 `origin_max`
   - 常温区最高分 -10
   - 热区最低分 = `origin_max`
   - 热区最高分 -10
3. 新内容进入热区

**降级流程**：
- 评分减少后，可能从热区转移到常温区（如果排名下降）
- 边界衰减自动实现流动，无需额外操作

**新兴热门保护**：
- 最近48小时内使用过的内容，即使评分不高，也可能进入新兴热区
- 新兴热区有保留席位（默认10个）

**删除规则**：
- 冷区超限时，删除最低评分的数据

### 4.5 边界衰减机制

```javascript
/**
 * 边界衰减：当新内容要进入热区时触发
 * 
 * 衰减流程：
 * 1. 保存常温区最高分 origin_max
 * 2. 常温区最高分 -10
 * 3. 热区最低分 = origin_max
 * 4. 热区最高分 -10
 * 5. 新内容进入热区
 * 
 * 优势：
 * - 计算量 O(1)，只计算边界值
 * - 自动流动：热区最低分流入常温区
 * - 分数平衡：热区和常温区最高分都会下降
 */
function boundaryDecay(hotItems, warmItems, newScore) {
  // 检查是否需要触发衰减
  if (hotItems.length === 0 || newScore <= hotItems[hotItems.length - 1].score) {
    return { hotItems, warmItems, triggered: false };
  }
  
  // 步骤1：保存常温区最高分
  const originMax = warmItems.length > 0 ? warmItems[0].score : 0;
  
  // 步骤2：常温区最高分 -10
  if (warmItems.length > 0) {
    warmItems[0].score = Math.max(0, warmItems[0].score - 10);
  }
  
  // 步骤3：热区最低分衰减到原常温区最高分
  const hotMinItem = hotItems[hotItems.length - 1];
  hotMinItem.score = originMax;
  
  // 步骤4：热区最高分 -10
  if (hotItems.length > 0) {
    hotItems[0].score = Math.max(0, hotItems[0].score - 10);
  }
  
  // 步骤5：新内容进入热区（由调用方处理）
  return { hotItems, warmItems, triggered: true, originMax };
}
```

## 5. 热门索引机制

### 5.1 热门索引定义

热门索引是位于热区的索引，优先返回给 Agent 查询。热区包含两类内容：
1. **历史高分内容**：评分高且排名靠前的内容
2. **新兴热门内容**：最近48小时内频繁使用的内容

### 5.2 热门索引返回规则

1. Agent 查询索引时，首先返回热区索引
2. 热区索引按最终评分降序排列
3. 如果热区索引不足，可补充常温区索引（可选）
4. 返回时包含分区标识（🔥[热]、🌡️[常温]、❄️[冷]）

### 5.3 热门索引更新机制

- **实时更新**：每次使用后更新评分，重新计算分区
- **批量重算**：定期（如每天）重新计算所有索引的分区
- **手动调整**：可通过管理脚本手动调整分区配置
- **新兴热门保护**：最近48小时内使用过的内容自动获得热区席位


```
       --id <id> [--score <score>] [--reset-times]

输入:
  --scope         项目隔离标识（必填）
  --type          数据类型：relation | keyword | index（必填）

  1. 读取当前评分和使用时间记录

  4. 重新计算分区（使用混合分区算法）
```

## 8. 异常处理

| 场景 | 行为 | 是否对外暴露 |
|------|------|-------------|
| 评分计算溢出 | 限制最大评分（如 9999），超过则不再增加 | 否 |
| 分区配置错误 | 使用默认配置，记录警告 | 是（警告） |
| 冷区超限 | 删除最低评分数据，记录删除数量 | 否（自动处理） |
| 评分数据损坏 | 重置评分为 0，清空 lastUsedTimes，记录警告 | 是（警告） |
| 并发评分更新 | 使用乐观锁机制，冲突时重试 | 是（错误） |
| 分区转移失败 | 记录错误，不影响正常使用 | 是（错误） |
| lastUsedTimes 数组损坏 | 重置为空数组，记录警告 | 是（警告） |
| 活跃度加成计算异常 | 返回默认加成系数 1.0，记录警告 | 否（自动恢复） |
| 时间戳格式错误 | 忽略该记录，继续处理其他记录 | 是（警告） |
| 新兴热门席位不足 | 按最后使用时间排序，优先保留最新的 | 否（自动处理） |

## 9. 性能考虑

### 9.1 评分计算性能

- **基础评分计算**：< 1ms（简单数学计算，最多10个时间戳）
- **时间衰减计算**：< 0.1ms（指数计算）
- **活跃度加成计算**：< 0.1ms（条件判断）
- **最终评分计算**：< 1ms（三部分计算总和）

### 9.2 分区计算性能

- **混合分区算法**：O(n log n)（排序） + O(n)（分区）
- **1000条数据分区**：约 5-10ms
- **10000条数据分区**：约 50-100ms

### 9.3 查询性能

- **热门索引查询**：< 5ms（内存排序）
- **分区过滤查询**：< 2ms（线性扫描）
- **统计信息计算**：< 10ms（遍历所有数据）

### 9.4 并发性能

- **乐观锁机制**：冲突时重试，最多重试 3 次
- **并发评分更新**：无锁设计，最终一致性
- **分区重算**：定期批量重算，避免频繁计算

### 9.5 存储性能

- **lastUsedTimes 数组**：最多存储10个时间戳，存储开销极小
- **评分数据文件**：每条数据约 200-500 字节
- **分区配置文件**：固定大小，约 1KB
