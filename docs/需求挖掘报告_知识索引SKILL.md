## 2. 需求澄清

### 2.1 需求形态（真实需求 / 还原后的需求）
**真实需求**：为 AI Agent 构建轻量级知识导航层，解决知识检索效率低和知识盲区问题

### 2.2 功能本质
为 AI Agent 构建一个轻量级的知识导航层，解决"AI 不知道项目有什么知识"和"热门知识重复检索浪费资源"两个核心问题

### 2.3 使用场景与角色
- 场景 1：**开发者** 在启动项目时，通过 Group 树索引快速了解项目知识结构，无需手动探索
- 场景 2：**AI Agent** 在辅助开发时，热门知识（<10ms）直接从本地 JSON 返回，避免重复语义检索
- 场景 3：**AI Agent** 遇到冷门知识时，通过关键词词云组装查询，走语义检索路径
- 场景 4：**AI Agent** 发现知识缺失时，主动暂停并引导用户补充，形成知识闭环
- 场景 5：**开发者** 导入外部知识库（如 docs/ 目录），通过预扫描+摘要向量化快速复用已有文档资产

**场景关联性**：场景 1-3 为知识查询场景，场景 4 为知识补充场景，场景 5 为知识导入场景，三者共同构成完整的知识管理闭环

### 2.4 用户角色
主要用户是**开发者**（知识管理员）和**AI Agent**（知识消费者）

### 2.5 核心痛点
1. **无导航入口**：AI 不知道项目有哪些知识域，只能通过试探性查询逐步了解
2. **重复检索成本**：热门知识每次都被重复检索，即使内容未变
3. **冷门知识不可见**：低频 Relation 在缓存中消失后，AI 完全不知道其存在
4. **索引维护手工**：没有系统化的索引管理工具，靠人工记忆 Group 结构

### 2.6 期望体验
- AI Agent 首次接触项目时，通过一次 Group 索引查询获得完整知识全景
- 热门知识秒级返回（<10ms），冷门知识通过语义检索发现
- 知识缺失时 AI 主动引导用户补充，形成知识闭环
- 外部知识库一键导入，快速复用已有文档资产
- 新兴热门内容能快速进入热区，避免历史霸占

### 2.7 深层动机
1. **提升 AI Agent 开发效率**：减少知识检索时间，让 AI 更快理解项目
2. **降低知识检索成本**：热门知识走本地路径，避免重复网络请求
3. **知识闭环**：AI 主动发现知识缺失并引导补充，避免知识盲区
4. **复用已有资产**：外部知识库导入，避免从零构建

### 2.8 非功能性需求
- 性能：热门知识 <10ms 响应，Group 树索引 <5ms
- 安全：scope 隔离、输入校验、乐观锁机制
- 可用性：零配置导入、渐进式读取由 AI 判断
- 数据完整性：自动同步机制、归档机制、数据备份

## 3. 根本性分析

### 3.1 核心问题
**AI Agent 在复杂项目中缺乏高效的知识导航机制，导致知识检索效率低下和知识盲区**

### 3.2 根因链
1. **直接原因**：AI Agent 不知道项目有哪些知识域，只能通过试探性查询逐步了解
2. **深层原因**：缺乏结构化的知识导航入口和智能缓存机制
3. **根本原因**：知识表示方式（层次化 Group 树）可能无法完全表达复杂项目中的网状关系

### 3.3 方案评估（情况 B + 理由）
**你描述的功能**：三层架构（Group树索引 → Relations缓存 → 本地KB）+ 双路径路由 + 评分机制 + 边界衰减 + 外部知识库导入

**我的判断**：**情况 B：方案部分有效**

**理由**：
1. **问题-方案匹配度**：高 - 方案直接针对知识导航和性能优化，对症下药
2. **症状 vs 根因**：方案在消除表面症状的同时，也触及了根本原因（导航入口、缓存机制）
3. **效果覆盖度**：高 - 覆盖了主要场景（首次查询、热门知识、冷门知识、知识缺失、外部导入）
4. **普适性**：中 - 针对代码开发场景深度优化，但核心思路可扩展
5. **副作用**：中 - 引入了评分机制和边界衰减，增加了系统复杂度

### 3.4 预期效果分析
- **核心场景覆盖度**：高（90%） - 覆盖了代码开发知识导航的主要场景
- **痛点解决程度**：高（85%） - 直接解决了导航入口、性能优化、知识闭环等痛点
- **用户体验提升**：
  - AI Agent：从"盲目搜索"到"精准导航"，热门知识秒级返回
  - 开发者：从"手动维护"到"自动管理"，外部知识库一键导入
- **潜在副作用**：
  - 评分机制和边界衰减增加了系统复杂度
  - 依赖 AI 判断渐进式读取，可能存在准确性风险
  - Group 树结构可能无法完全表达复杂项目的网状关系

### 3.5 建议（短期/长期/如坚持原方案）
- **短期**：当前方案可行，能有效解决知识导航和性能优化问题
- **长期/根本**：
  1. 探索更灵活的知识表示方式（如知识图谱），支持复杂关系表达
  2. 建立 AI 判断的校验机制，确保渐进式读取的准确性
  3. 考虑知识自动发现和索引构建，减少人工维护成本
- **如果坚持原方案**：需要接受 Group 树结构的局限性，对于复杂关系可能需要额外处理

### 4.6 非功能性约束

1. **性能约束**：
   - 热门知识响应时间 <10ms
   - Group 树索引读取延迟 <5ms
   - 评分计算延迟 <1ms

2. **安全约束**：
   - scope 隔离：不同 scope 之间通过文件系统路径物理隔离

## 5. 潜在风险与注意事项

1. **评分机制准确性风险**：
   - 风险：评分算法可能无法准确反映知识热门程度
   - 预案：通过实际使用数据调整算法参数，支持配置化

2. **边界衰减参数风险**：
   - 风险：-10 分值可能不合适，导致分数剧烈波动
   - 预案：通过模拟数据测试不同分值的影响，支持配置化

3. **新兴热区席位不足风险**：
   - 风险：10 个保留席位可能不够，导致新兴热门无法进入热区
   - 预案：根据项目规模动态调整，支持配置化

4. **AI 判断准确性风险**：
   - 风险：AI 判断渐进式读取的准确性可能不足
   - 预案：建立校验机制，必要时人工干预

5. **Group 树结构局限性风险**：
   - 风险：层次化结构可能无法完全表达复杂项目的网状关系
   - 预案：对于复杂关系，可通过关键词词云补充语义信息

6. **并发写入风险**：
   - 风险：高并发场景下文件覆盖导致数据丢失
   - 预案：使用乐观锁机制，冲突时返回错误提示重试

7. **数据一致性风险**：
   - 风险：本地 KB 与记忆系统内容不一致
   - 预案：采用自动同步机制，当 Agent 发现记忆不存在时自动同步
