# 设计决策记录

## 1. 架构决策

### 1.1 三层文件系统架构
- **决策**：Group树索引 → Relations缓存 → 本地KB
- **理由**：AI首次接触项目时可通过Group索引一次获取完整知识导航结构，热门知识走本地JSON快速路径（<10ms），冷门知识退化为关键词词云引导语义检索
- **被否决方案**：直接使用记忆系统（每次走网络请求）

### 1.2 Scope隔离机制
- **决策**：全链路显式 `--scope` 参数
- **理由**：与MCP启动参数一致，无需额外配置，避免隐式传递易出错
- **被否决方案**：环境变量传递

### 1.3 本地KB存储格式
- **决策**：JSON文件 + 目录分层
- **理由**：零依赖、可Git版本控制、TS脚本一行解析
- **被否决方案**：LanceDB（过度设计）、SQLite（引入额外依赖）

### 1.4 模块信息格式
- **决策**：Markdown纯文本 + 代码定位符
- **理由**：LLM天然理解、人可直接读写；代码路径帮助AI快速定位源码
- **被否决方案**：纯结构化JSON（字段模板僵化）、纯文本无代码路径（AI需额外搜索定位）

## 2. 评分机制决策

### 2.1 评分算法
- **决策**：基于使用密度的复合评分（基础分 × 活跃度加成）
- **理由**：更准确反映使用频率，避免简单+1计数被临时高频调用误导
- **被否决方案**：简单+1计数

### 2.2 冷热分区机制
- **决策**：三区机制（新兴热区/历史热区/常温区/冷区）
- **理由**：新兴热区有保留席位，历史热区按评分排序，自动流动
- **被否决方案**：单一分区（无法区分使用频率）

### 2.3 边界衰减机制
- **决策**：只在内容进入热区时触发衰减，计算量O(1)
- **理由**：避免实时衰减的计算量大问题，性能最优
- **被否决方案**：实时衰减（计算量大，性能差）

### 2.4 新兴热区保留席位
- **决策**：最近48小时内使用过的内容有保留席位（默认10个）
- **理由**：解决历史霸占热区问题，新内容能快速进入热区
- **被否决方案**：全量返回（信息过载）

## 3. 数据管理决策

### 3.1 淘汰策略
- **决策**：最低评分淘汰 + 退化为关键词保留在词云
- **理由**：不丢失语义锚点，仍可通过关键词回退检索
- **被否决方案**：直接删除（数据彻底丢失）

### 3.2 关键词规则
- **决策**：禁止代码符号，仅自然语言
- **理由**：代码路径/类名/方法名会引入噪音，降低语义检索精度
- **被否决方案**：允许代码符号（检索时路径信息干扰向量匹配）

### 3.3 导入知识处理
- **决策**：仅摘要向量化，原文只存本地KB
- **理由**：摘要语义密度高、匹配质量好、成本极低；原文中代码/配置会稀释语义信号
- **被否决方案**：原文向量化（成本高且语义噪音大）

### 3.4 导入根节点隔离
- **决策**：独立根节点隔离
- **理由**：避免自建知识与导入知识混合，Group路径天然隔离
- **被否决方案**：共享根节点（两类知识混在一起）

## 4. 技术选型决策

### 4.1 查询脚本实现
- **决策**：JS脚本（`.mjs`）
- **理由**：项目已有jiti基础设施，直接运行；JS生态成熟
- **被否决方案**：TS需额外编译步骤

### 4.2 模块检索脚本
- **决策**：TS脚本（`.ts`）
- **理由**：可用jiti直接执行，类型安全
- **被否决方案**：JS无类型，容易传错参数

### 4.3 数据一致性机制
- **决策**：自动同步机制
- **理由**：当Agent发现记忆不存在时自动同步，无需主动检查，实际场景中更高效
- **被否决方案**：一致性检查（增加复杂度）

### 4.4 并发导入处理
- **决策**：拆分合并机制
- **理由**：每个Agent单独创建索引，完成后合并，避免索引混乱
- **被否决方案**：直接并发写入（可能导致索引冲突和数据不一致）

## 5. 风险控制决策

### 5.1 超大文件处理
- **决策**：直接删除
- **理由**：文档数据不应存在超大文件，读取会浪费巨量Token
- **被否决方案**：限制读取（仍可能产生大文件）

### 5.2 特殊字符处理
- **决策**：直接删除
- **理由**：最简单有效，避免复杂转义逻辑
- **被否决方案**：转义处理（增加复杂度，可能引入新问题）

### 5.3 数据备份机制
- **决策**：自动备份机制
- **理由**：避免知识库丢失后重新创建，提供恢复能力
- **被否决方案**：无备份（数据丢失风险高）

### 5.4 乐观锁机制
- **决策**：文件修改时间戳检测
- **理由**：避免高并发产生数据问题，简单有效
- **被否决方案**：无锁机制（可能导致数据覆盖丢失）
